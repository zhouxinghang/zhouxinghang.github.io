reactor 线程模型及在 netty 中的应用

Posted on 2021-11-26 In java进阶 Views: Disqus:
reactor 是事件驱动模型，netty 基于 nio 的事件驱动线程模型，实现高并发。

IO 模型

BIO

NIO

阻塞 I/O 模型

用户进程调用 recvfrom，内核没有数据会返回 error（立即返回），有数据会阻塞将数据拷贝到用户态并返回 OK

所以，nonblocking IO的特点是用户进程需要不断的主动询问kernel数据好了没有。

I/O 多路复用模型

上述轮训 recvfrom，效率较低。这里使用 select/epoll 系统调用，可以监听多个 socker fd。调用 select 阻塞等待返回有 I/O 事件的 fd。然后再调用 recvfrom 获取数据。

select/epoll 的优势在于多数 I/O 连接。当 I/O 连接很少时，使用 BIO 性能更高（只需要一个 recvfrom 命令）

AIO

用户进程向内核注册 I/O 事件，内核会等待数据准备完成，然后将数据拷贝到用户内存，最后会通知用户进程（发送 signal）

reactor 线程模型

reactor 类似于 dispatch 的意思，是一个事件驱动模型，应用业务向一个中间人（dispatcher）注册一个回调（event handler），当 I/O 就绪后，dispatcher 通知 event handler 进行处理

reactor 多线程模型

mainReactor 监听 accpet 事件，client 连接成功后，将连接转发给 subReactor。subReactor 监听读写事件（包括完成读写操作）

Acceptor 接收到客户端 TCP 连接请求处理完成后（可能包含接入认证等），将新创建的 SocketChannel 注册到 IO 线程池（subReactor 线程池）的某个 IO 线程上，由它负责 SocketChannel 的读写和编解码工作。Acceptor 线程池仅仅只用于客户端的登陆、握手和安全认证，一旦链路建立成功，就将链路注册到后端 subReactor 线程池的 IO 线程上，由 IO 线程负责后续的 IO 操作。

mainReactor 一般单线程就够了，subReactor 是多线程

netty 线程模型

同 reactoc 多线程模型，Netty里对应mainReactor的角色叫做“Boss”，而对应subReactor的角色叫做”Worker”

在Netty 4.0之后，作者觉得NioSelector这个叫法，以及区分NioBoss和NioWorker的做法稍微繁琐了点，干脆就将这些合并成了NioEventLoop，从此这两个角色就不做区分了。

上图是 Netty4 的 EventLoop 处理流程，IO 读写和业务处理都在 worker 线程中

如果业务处理耗时长，会影响 worker 线程的整体并发，需要异步处理业务流程
