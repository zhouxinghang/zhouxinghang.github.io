阅读领域驱动设计书籍的个人总结

概述

领域驱动设计（Domain-Driven Design，简称DDD）起源于2004年Eric Evans出版《领域驱动设计》，用于应对复杂需求的软件开发。随着近几年微服务架构的普及，DDD 又火起来了

传统开发中，先根据业务需求设计好数据库 ER 图，然后实现代码。这种方式业务逻辑与技术细节精密耦合不利于后续的需求扩展。DDD 中先进行业务领域建模，再实现代码，让业务逻辑不依赖于技术细节（技术细节越往后决定越好）。

本篇文章是《领域驱动设计模式、原理与时间》读书笔记总结，重点参数领域驱动设计思想，有疑问地方欢迎讨论。文中的业务场景以会议预定系统为例，详见 Meetup.com

DDD 领域概念

名称定义举例
实体有身份标识，一般对应业务对象，封装属性（不公开），公开领域行为会议是个实体，表示一个唯一被创建的会议，包含会议成员、地点、会议主题等属性，提供创建会议、添加成员、添加评论等领域行为
值对象富含行为，无身份，不可变，参数校验会议室地址是个值对象，会校验地址合法性，能清晰的表示业务属性。 手机号值对象封装 long 基元属性，有校验参数，业务语义
聚合一组业务紧密管理的实体和值对象，是数据修改和持久化的基本单元 聚合内保持强一致性，聚合间保持最终一致性会议是个聚合，MeetingGroup 实体是该聚合的根实体，组成员、组地点等是附属聚合的值对象。对领域外公开创建会议，添加成员的操作
存储库封装资源（不局限于数据库、文件、网络存储），只公开需要的操作。领域模型映射到持久化存储会议聚合的存储、更新、查询都由存储库代理完成
领域事件解耦不同领域的依赖，B 领域需要响应处理 A 领域的操作。同一个微服务中可以是同步事件MeetingGroup 申请，申请通过后，会发送事件给会议域，会议域创建 MeetingGroup。解耦会议域与管理员域
领域服务领域服务没有任何属性或数据，只是一个领域行为或动作，不适合放在任何聚合内的逻辑行为比如领域内通用的校验逻辑
应用服务应用服务对应到一个具体业务场景，通过编排聚合、资源库、领域事件、外部适配接口、领域服务来完成Meeting 创建场景，调用存储库获取 MeetingGroup，调用 MeetingGroup 创建会议，通知会议创建事件（推荐放在领域中）

战略模式——在限界上下文通信

战略模式看似很虚，but 是最重要的。与领域专家、PM一起讨论提炼问题域，编写 UL。划分领域和限界上下文（Bouded Context），并确定上下文之间的关系。

什么是领域、子域和限界上下文？

领域代表现实世界的特定问题和解决方案的集合。限界上下文（Bouded Context）定义了模型的适用范围，是一个语言边界，可以是一个微服务。自主拥有展现-领域逻辑-数据库的完整代码堆栈。

子域是问题空间的概念，限界上下文是解空间的概念。所以限界上下文一定可以对应上一个子域（子域可以对应多个上下文）。一个限界上下文就是一个语言边界，可以是一个微服务。自主拥有展现-领域逻辑-数据库的完整代码堆栈。

如何划分限界上下文？

根据业务概念来划分，实际需要参考团队粒度、技术特点做裁剪

领域概念只在限界上下文内有意义。比如同样是订单，在不同的上下文表示不同的概念

如何区分值对象与实体

值对象只是问题域的属性和特征。实体具有唯一身份标识，具有行为和生命周期。值对象和传统的 java bean 类似，区别是：不可变，有参数校验。

以图书馆系统为例，图书具有唯一标识是一个实体，有出租、归还等行为。图书的 ISBN 号是图书的 ID 属性是一个值对象。为什么不把 ISBN 定义为 string 简单字段？以为 ISBN 有一些隐藏属性，包括 ISBN 的类型和合法性校验。

为什么需要聚合

一些对象具有不可变条件，以采购系统为例，当采购请求被通过后，不能修改采购项。采购项与采购请求两个实体具有不变条件，放在一个聚合中。

为保证聚合内的强一致性，外界只能引用聚合根，不能引用聚合内部对象。如有必要，聚合返回内部对象的副本给外界。

领域模型到底是什么

描述业务逻辑领域概念，偏向代码实现的领域对象。我将其理解为领域层的整个代码实现

DDD 将业务分为 4 层：UI、应用、领域、基础。其核心是高度提纯、通用、少变化的领域层。领域层对外暴露业务行为接口，驱动应用层和基础层的设计，是谓“领域驱动设计”。
