kafka最佳实践

Posted on 2022-01-05 In 中间件 Views: Disqus:
kafka 重要参数 ascks retries 使用。rebalance 规避策略

KafkaProducer

消息相关

一些参数

ascks

0,1.-1。从左到右代表了更严格的消息持久化保障。配合 retries 参数一起使用

0 只保证消息从客户端发出，不处理服务端返回的任何响应

1 表示只保证消息能成功写入分区的 leader，不保证所有副本都保留这份数据

-1 保证所有的副本和 leader 都成功写入

retries

表示在服务端返回异常的时候客户端重试次数。

例如业务反馈的 NotLeaderForPartitionException 便是在副本切换时发生的短暂异常。这是只要使用客户端重试策略即可

buffer.memory & batch.size & linger.ms

一般 kafka 客户端调用 send 方法后，消息会暂存在 buffer 中，最后批量发送给 kafka broker

用户线程发送消息会暂存在 buffer （java是堆内内存）中，形成一个个 Batch，每个 Batch 包含多条消息。然后 Sender 线程会把多个Batch 打包成一个 Request 发送到 kafka broker。如下图

buffer.memory 决定内存缓冲大小，设置过小会阻塞用户线程

batch.size 决定每个 Batch 的大小，决定每次发送给 broker 的数据包大小。太大了，会导致数据此次不能被发送

linger.ms 消息最大缓存时间，如果 Batch 未凑满到了 linger 时间也会发送给 broker

KafkaConsumer

消费 lag 的一些解决办法

  1. 增加消息并发，consumer 扩容
  2. 避免 rebalance
    1. 升级服务时增加并发度。并发度小，会持续引起 rebalance
    2. java 客户端 gc 优化，gc 过长会导致心跳超时，从而引起 rebalance
    3. 优化减少每次 poll 的时间，就能保证心跳的及时发送
      1. 通过 max.poll.records 参数，减少每次 poll 消息的大小
      2. 优化业务逻辑，减少 poll 处理事件。比如 poll 之后异步处理

适当减少提交 offset 的频率

kafka comsumer 支持自动提交 offset，将整个 consumer group 的消费进度保存在 ZK（就版本） 或 kafka 中。

一些开源组件配置了不合理的提交 offset 间隔导致频繁提交 offset，会给 kafka broker 带来很大压力。可以调整 auto.commit.interval.ms 参数来降低提交评率
